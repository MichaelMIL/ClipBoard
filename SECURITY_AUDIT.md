### H3. Key-replace race in `ClipboardPersistenceCrypto.storeKeyInKeychain` вЬЕ

| | |
|---|---|
| Found by | `security-auditor` |
| Fixed by | `security-architect` (decision) + `coder` (apply) |
| Status | вЬЕ Fixed in `ClipboardAppLib/ClipboardPersistenceCrypto.swift` |
| Verification | `readKeyDataFromKeychain` throws `keychainNotFound` on `errSecItemNotFound` and `keychainAccess(status)` otherwise. `loadOrCreateSymmetricKey` only generates on `keychainNotFound`. ### H4. AES-GCM seal uses no AAD / no file-role binding вЬЕ

| | |
|---|---|
| Found by | `security-architect` |
| Fixed by | `coder` |
| Status | вЬЕ Fixed in `ClipboardAppLib/ClipboardPersistenceCrypto.swift` |
| Verification | New `CLP2` envelope uses `AES.GCM.seal(_:using:authenticating:)` with `Data("CLP2|<role>|v1")` as AAD per file role (`history` / `favorites`). `unwrap` opens CLP2 with the matching AAD. Swap or downgrade across roles now fails authentication. CLP1 still loads (no AAD), but `formatMismatch(_:)` in `ClipboardHistoryStore` marks any non-CLP2 ciphertext for re-encryption on next save. |

## New findings surfaced during the remediation pass

### R1. `try? setAttributes` swallows permission-set failure вЪ†пЄП

| | |
|---|---|
| Found by | `code-analyzer` |
| Severity | Low |
| Status | вЪ†пЄП Open |

In `ClipboardHistoryStore.saveCollection` and `supportDirectoryURL`, the calls to `FileManager.default.setAttributes([.posixPermissions: вА¶], вА¶)` use `try?`. If for some reason permissions cannot be tightened (e.g. ACLs prevent it), the file remains world-readable and we silently move on. Recommend: catch the error, log via `Logger`, and consider a one-time alert.

### R2. Brief world-readable window between atomic write and `setAttributes` вЪ†пЄП

| | |
|---|---|
| Found by | `security-architect` |
| Severity | Low |
| Status | вЪ†пЄП Open |

`Data.write(to:options: .atomic)` writes a temp file under default umask (typically `0644`) and then renames into place. The follow-up `setAttributes` tightens to `0o600` immediately, but there is a sub-millisecond window where another local user could `cat` the renamed file. Tightening would require switching to `FileManager.createFile(atPath:contents:attributes:)` with `[.posixPermissions: 0o600]` against a temp name followed by an explicit `replaceItemAt`. Lower-priority because the window is tiny relative to the previous "permanently 0644" state.

### R3. Quarantined files are logged but not surfaced to the user вЪ†пЄП

| | |
|---|---|
| Found by | `reviewer` |
| Severity | Low |
| Status | вЪ†пЄП Open |

The original L10 recommendation included "surface a one-time user alert." The current implementation only logs via `Logger`. A user whose history is unreadable would see history reset to empty and never know a quarantine occurred. Recommend a banner via `ClipboardCopyNotifier` or a sticky `NSAlert` on next launch.

### R4. Encrypted output uses `JSONEncoder.outputFormatting = [.prettyPrinted, вА¶]` вЪ†пЄП

| | |
|---|---|
| Found by | `reviewer` |
| Severity | Negligible |
| Status | вЪ†пЄП Open |

When the file is encrypted, pretty-printing adds bytes inside the ciphertext for no human benefit. Switch to `[.sortedKeys]` only when `encrypted == true` to shrink encrypted blobs by ~20%. Not security-relevant; quality-of-life only.
